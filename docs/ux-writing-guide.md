# MISHIRU UXライティング基準

画面の文章は、ブランドやシステムを説明するためではありません。利用者が意味を推測せず、目的を達成できるように書きます。

## 0. 書く前に決めること

- この画面を読む人は、いま何をしようとしているか。
- その人が最初に知りたいことは何か。
- 読んだ後に、どの行動を選べれば成功か。
- 不安・迷い・待ち時間のうち、いま減らすべき負担は何か。

文章は `目的 → 読み手 → 次の行動 → 必要な情報` の順で設計します。

## 1. 結論を先に書く

- 見出しだけで、画面の目的がわかるようにする。
- 説明は「何ができるか」→「どう操作するか」の順にする。
- 重要な内容は3点以内に絞る。補足は開閉できる場所へ移す。
- 背景よりも、利用者にとっての結果を先に書く。

## 2. 短く、具体的に書く

- 一文では一つだけ伝える。
- 「〜することができます」は「〜できます」にする。
- 「こちら」「詳細」「次へ」だけのボタンは使わない。
- 漢字が続くときは、普段使う言葉をひらがなにする。
- 画面を閉じ、数秒後に読み直しても意味が取れる文にする。

## 3. ボタンには、押した後の結果を書く

- 良い例：`問いの候補をつくる`、`もう一度読み込む`、`研究プランを保存`
- 避ける例：`実行`、`送信`、`再試行`、`OK`
- 読み手の行動を動詞で書く。サービス側の都合は書かない。
- ユーザーの現在地より先の決断を迫らない。まず「見る」「比べる」「確認する」を案内する。

## 4. 初めて使う人の言葉を優先する

- 専門語は必要な場所だけで使い、その場で意味を添える。
- `RQ` は一般向けの画面では「研究の問い（RQ）」と書く。
- `バージョン` は「別の案」「変更履歴」と書く。
- 装飾目的の英語は、操作や理解に必要な場所では日本語へ置き換える。
- AIの処理名、品質検査、フォールバックなど、運営側の言葉をそのまま見せない。

## 5. 状態と次の行動をセットで伝える

- 読み込み中：何をしているか、目安時間を書く。
- 空の状態：なぜ空か、何をすれば増えるかを書く。
- エラー：何ができなかったか、次に押すボタンを書く。
- 完了：何が保存・更新されたかを書く。
- 待機中：何を処理しているかを書く。根拠のない所要時間は約束しない。

## 6. 場面ごとに機能と情緒の比率を変える

| 場面 | 優先すること |
|---|---|
| 操作・認証・削除・エラー | 正確さと次の行動 |
| 検索・比較・問いづくり | 分かりやすさと心理的な進みやすさ |
| ブランドの見出し | 理解を妨げない範囲の情緒 |
| 論文・研究プラン | 正確さを保った平易な説明 |

## 7. MISHIRU内の呼び方をそろえる

| 目的 | 表示名 |
| --- | --- |
| 保存内容を見る | 保存したもの |
| 研究の入口を見る | 問いを見る／問いのカード |
| 関心を分析する | 関心を整理 |
| リサーチクエスチョンを作る | 問いをつくる |
| 研究プロジェクトを管理する | 研究プラン |
| 研究室候補と共有資料を見る | 相談先を探す |

## 公開前の確認

- 見出しだけで、画面の目的がわかるか。
- ボタンだけを拾い読みして、押した後がわかるか。
- エラーだけを読んで、次にできることがわかるか。
- 専門語を知らない人でも、意味を推測せず読めるか。
- 同じ内容を二度説明していないか。
- AI生成文だけ、急に難しい表現へ戻っていないか。
