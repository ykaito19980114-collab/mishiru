# MISHIRU UXライティング基準

画面の文章は、ブランドを説明するためではなく、利用者が次の行動を迷わず選ぶために書きます。

## 1. 結論を先に書く

- 見出しだけで、画面の目的がわかるようにする。
- 説明は「何ができるか」→「どう操作するか」の順にする。
- 重要な内容は3点以内に絞る。補足は開閉できる場所へ移す。

## 2. 短く、具体的に書く

- 一文では一つだけ伝える。
- 「〜することができます」は「〜できます」にする。
- 「こちら」「詳細」「次へ」だけのボタンは使わない。
- 漢字が続くときは、普段使う言葉をひらがなにする。

## 3. ボタンには、押した後の結果を書く

- 良い例：`問いの候補をつくる`、`もう一度読み込む`、`研究プランを保存`
- 避ける例：`実行`、`送信`、`再試行`、`OK`
- 読み手の行動を動詞で書く。サービス側の都合は書かない。

## 4. 初めて使う人の言葉を優先する

- 専門語は必要な場所だけで使い、その場で意味を添える。
- `RQ` は一般向けの画面では「研究の問い（RQ）」と書く。
- `バージョン` は「別の案」「変更履歴」と書く。
- 装飾目的の英語は、操作や理解に必要な場所では日本語へ置き換える。

## 5. 状態と次の行動をセットで伝える

- 読み込み中：何をしているか、目安時間を書く。
- 空の状態：なぜ空か、何をすれば増えるかを書く。
- エラー：何ができなかったか、次に押すボタンを書く。
- 完了：何が保存・更新されたかを書く。

## 6. MISHIRU内の呼び方をそろえる

| 目的 | 表示名 |
| --- | --- |
| 保存内容を見る | 保存したもの |
| 関心を分析する | 関心を整理 |
| リサーチクエスチョンを作る | 問いをつくる |
| 研究プロジェクトを管理する | 研究プラン |
| 研究室候補と共有資料を見る | 相談先を探す |

公開前は、ボタンだけを拾い読みして行き先がわかるか、エラー表示だけで次の行動がわかるかを確認します。
