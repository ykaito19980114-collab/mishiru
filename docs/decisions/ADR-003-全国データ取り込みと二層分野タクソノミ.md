# ADR-003: 全国研究室データ(19,785件)の取り込みと二層の分野タクソノミ

- 日付: 2026-07-04 ／ 状態: 採択
- 関連: docs/02 §4-5、docs/03 §2・§5・§6、ユーザー提供の別チャット設計（派生列・大学マスタ・分野分類）

## 文脈
プロトタイプは118研究室のサンプルで動いていたが、実データとして全国100大学・19,785研究室のCSV（第1回〜第46回統合）が提供された。別チャットで「派生列追加・100大学の地域マスタ・12分野自動分類」の最適化が設計済み。これを本リポジトリのデータパイプラインとして実装し、サービスに反映する。

## 決定
1. **正規化はビルド時パイプライン**（`scripts/convert-national.ts`）で行い、`data/labs.json`(24MB)を生成する。Supabase移行時は同じ派生ロジックをSQL（supabase/schema.sql のコメント参照）に写す。CSVを一次ソースとし、DBは段階導入（ADR-002を踏襲）。
2. **派生列**：研究科/専攻の分割、pi_name/pi_title抽出、member_count、keyword_tags、official_url正規化（"要確認"等→null）、has_url、大学マスタ由来の prefecture/region/university_type。
3. **大学マスタ**（`shared/universities.ts`）：100大学すべてに設置区分・都道府県・地域を付与。convert時に全件突合し、未マッピングがあればビルドを止める（漏れゼロを保証）。
4. **分野は二層タクソノミ**：
   - **大分類12（`shared/fields.ts` の field_major）**：フィルタUI用。キーワード＋専攻＋研究室名からルール分類。実データで未分類1.8%（目標2%以下）。
   - **細分15（`shared/taxonomy.ts` の area_tags）**：カード⇄研究室マッチング用。既存を維持。
   両者を併存させる（フィルタは粗く網羅的に、マッチングは細かく意味的に）。

## 理由
- 2万件の完全正規化（教員を別テーブル化）は移行コストが見合わない。別チャット結論どおり「1メインテーブル＋派生列＋軽量マスタ」の中間解を採る。
- 分野を二層にするのは、フィルタ（母集団を大きく絞る）とマッチング（興味カードとの意味的近さ）で必要な粒度が異なるため。
- URL未登録37%（7,350件）は営業リストの資産。has_urlで抽出可能にし、管理画面「研究室データ」タブに出す（BR-06の営業初日リスト）。

## 品質の但し書き
- 分野分類はキーワード単純一致のため、まれに誤分類がある（例：「環境エネルギー」が物理寄り）。フィルタ用途には十分（未分類1.8%）。気になる研究室は将来 field_major を手修正可能にする。
- 地域・設置区分は本部所在地基準（分校・キャンパス分散は考慮しない）。

## 影響
- 論文検索は自前で持たず、研究室ページから外部DB（researchmap/CiNii/KAKEN/Google Scholar）へ教員名で橋渡しする（別チャットのLv.1方向B）。論文DBの巨人と正面衝突せず、着地点（研究室のリアル）で差別化する方針をdocs/01のポジショニングに追加。
- `data/labs.json` が24MBに。git管理はするが、将来Supabase移行で軽量化。起動時のJSON parseは実測数百ms以内で許容。
